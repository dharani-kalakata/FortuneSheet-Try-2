The `webview-src/` files run in a **browser iframe** (the webview panel).

## 2. Every File Explained

### `package.json`

**What:** The manifest file that tells VS Code everything about this extension.

**Key sections:**
- `"main": "./src/extension.js"` → entry point when VS Code loads the extension
- `"contributes.customEditors"` → registers our editor for `*.json` files with `"priority": "option"` (VS Code offers it as an option, not the default)
- `"dependencies"` → FortuneSheet v1.0.4 + React 18 (used by the webview)
- `"devDependencies"` → webpack, babel, loaders (used only at build time)

---

### `webpack.config.js`

**What:** Tells webpack how to bundle the webview source code into a single `dist/webview/bundle.js`.

**Why needed:** The webview runs in a browser context. It can't use `require()` or import from `node_modules` directly. Webpack resolves all imports and produces one file that includes React, FortuneSheet, CSS, and our code.

**Key settings:**
- Entry: `./webview-src/main.jsx`
- Output: `dist/webview/bundle.js`
- `babel-loader` → converts JSX syntax into plain JavaScript
- `style-loader` + `css-loader` → injects CSS (from FortuneSheet) directly into the JS bundle

---

### `.vscodeignore`

**What:** When you package the extension into a `.vsix` file (for distribution), this tells `vsce` which files to exclude. Keeps the package small.

---

### `.gitignore`

**What:** Standard Git ignore file. See [Section 13](#13-gitignore-explained).

---

### `src/extension.js`

**What:** The activation entry point.

**What it does:**
1. VS Code calls `activate(context)` when the extension is needed
2. It calls `JsonEditorProvider.register(context)` to register the custom editor
3. Returns a disposable that VS Code uses for cleanup

**Why it exists:** VS Code requires a `main` entry point. This file is intentionally thin — all logic lives in `jsonEditorProvider.js`.

---

### `src/jsonEditorProvider.js`

**What:** The heart of the extension. Implements `CustomTextEditorProvider`.

**Why it exists:** This is the bridge between VS Code's text document system and the FortuneSheet webview. It handles:
- Loading JSON from the document
- Converting JSON to spreadsheet data (via `jsonConverter`)
- Creating and managing the webview
- Receiving edits from the webview
- Applying edits back to the document
- Validation on edit
- Warning on save
- Syncing undo/redo

**See [Section 3](#3-every-function-explained) for function-by-function breakdown.**

---

### `src/jsonConverter.js`

**What:** Pure data transformation module. No VS Code or DOM dependencies.

**Two directions:**
1. `jsonToSheetData(json)` → JSON object → FortuneSheet celldata + type map
2. `sheetDataToJson(values, typeMap, ...)` → 2-D value grid → JSON object (with types restored)

**Why it exists:** Keeps data conversion logic separate from VS Code plumbing. This module can be unit-tested independently.

---

### `src/jsonValidator.js`

**What:** Validates JSON rows against business rules before saving.

**Why it exists:** Catches data-entry mistakes early. Shows clear error messages with row and field references. See [Section 7](#7-validation-logic-detailed) for the complete rule breakdown.

---

### `webview-src/main.jsx`

**What:** The thinnest possible React entry point.

**What it does:**
1. Imports the theme helper (injects CSS)
2. Renders `<SpreadsheetEditor />` into the `#root` div

**Why it's thin:** Keeps concerns separated. The entry point only bootstraps; all logic is in components/hooks.

---

### `webview-src/components/SpreadsheetEditor.jsx`

**What:** The main React component. Renders FortuneSheet and handles communication.

**Responsibilities:**
- Renders `<Workbook>` from FortuneSheet with custom toolbar
- Handles `onChange` events from FortuneSheet
- Extracts cell values into a 2-D array
- Sends edits to the extension via `postMessage`
- Displays validation error banner
- Manages data versioning for undo/redo re-renders

---

### `webview-src/hooks/useVscodeMessaging.js`

**What:** A React hook that encapsulates all `postMessage` communication.

**What it does:**
1. Calls `acquireVsCodeApi()` once (VS Code requirement)
2. Listens for `message` events from the extension
3. Routes messages to the appropriate handler (`onLoadData`, `onValidationErrors`)
4. Sends a `ready` message on mount so the extension knows when to send data
5. Exposes a `postMessage` function for sending messages back

**Why a hook:** Keeps messaging logic out of the component. If you add new message types later, you only edit this file.

---

### `webview-src/utils/themeHelper.js`

**What:** A CSS injection module that makes FortuneSheet respect VS Code's color theme.

**Why it exists:** FortuneSheet ships with hardcoded white/light-gray colors. Without these overrides, the spreadsheet is unreadable in VS Code's dark theme. This file:
- Uses VS Code CSS custom properties (`--vscode-editor-background`, `--vscode-editor-foreground`, etc.)
- Overrides cell backgrounds, text colors, toolbar, formula bar, scrollbars, dropdowns, filter overlays, modal dialogs
- Uses `!important` to ensure our styles win over FortuneSheet's defaults
- Automatically adapts to light/dark/high-contrast themes

## 3. Every Function Explained

### `src/extension.js`

#### `activate(context)`
- **What:** Called by VS Code when the extension activates
- **Input:** `context` — VS Code's ExtensionContext (provides extensionPath, subscriptions, etc.)
- **Output:** None (registers the editor provider as a side effect)
- **Why:** Required by VS Code extension API

#### `deactivate()`
- **What:** Called when the extension deactivates
- **Input/Output:** None
- **Why:** Required by VS Code (currently a no-op; cleanup is handled by disposables)

---

### `src/jsonEditorProvider.js`

#### `JsonEditorProvider.register(context)`
- **What:** Static factory that creates a provider instance and registers it with VS Code
- **Input:** `context` — ExtensionContext
- **Output:** A `Disposable` (for cleanup)
- **Why:** Encapsulates registration logic; called once during activation

#### `resolveCustomTextEditor(document, webviewPanel)`
- **What:** Called by VS Code every time a `.json` file is opened with this editor
- **Input:**
  - `document` — the VS Code TextDocument (the `.json` file's content)
  - `webviewPanel` — the WebviewPanel where we render our UI
- **Output:** None (sets up the webview and message handlers as side effects)
- **Why:** This is the core lifecycle method of `CustomTextEditorProvider`

**Inside this function, several closures are defined:**

#### `loadDocument()` (closure)
- **What:** Reads the TextDocument, parses JSON, converts to sheet data, sends to webview
- **Input:** None (reads from `document` via closure)
- **Output:** None (sends `loadData` message to webview)
- **Why:** Called on initial open AND on undo/redo (document changed externally)
- **Critical:** Does NOT modify the document. Only reads and sends.

#### `_getHtml(webview)`
- **What:** Generates the HTML skeleton for the webview
- **Input:** `webview` — the Webview object (needed for URI conversion and CSP)
- **Output:** HTML string
- **Why:** The webview is a sandboxed iframe; we provide its HTML content
- **Key details:** Includes Content Security Policy, nonce-based script loading, VS Code theme CSS variables

#### `_getNonce()`
- **What:** Generates a random 32-character string
- **Input:** None
- **Output:** String (e.g., `"aB3xK9mQ..."`)
- **Why:** Used in Content Security Policy to allow only our specific script to run (prevents XSS)

---

### `src/jsonConverter.js`

#### `getValueType(value)`
- **What:** Returns a type tag string for any JS value
- **Input:** Any JS value (`null`, `42`, `"hello"`, `true`, `[]`, `{}`)
- **Output:** One of: `'null'`, `'array'`, `'object'`, `'string'`, `'number'`, `'boolean'`
- **Why:** We need to remember the original type of every cell so we can restore it on save

#### `valueToDisplay(value)`
- **What:** Converts any JS value to a string for spreadsheet display
- **Input:** Any JS value
- **Output:** String (e.g., `null` → `""`, `[1,2]` → `"[1,2]"`, `true` → `"true"`)
- **Why:** FortuneSheet cells display strings; arrays/objects need to be stringified

#### `jsonToSheetData(json)`
- **What:** The main conversion function — JSON → FortuneSheet format
- **Input:** Parsed JSON (object or array)
- **Output:** `{ sheetData, typeMap, structure, keys, headers }`
  - `sheetData` — array of sheet objects for FortuneSheet's `data` prop
  - `typeMap` — object mapping `"row_col"` → original type (e.g., `"1_3": "number"`)
  - `structure` — `'array'` | `'object-of-objects'` | `'object'`
  - `keys` — original top-level keys (for object-of-objects, preserves order)
  - `headers` — column header names
- **Why:** FortuneSheet needs data in a specific format (celldata array with row/col/value objects)

**How it detects structure:**
```
Array?               → structure = 'array', each element = one row
Object of objects?   → structure = 'object-of-objects', keys become _key column
Simple object?       → structure = 'object', single row
```

#### `emptySheet(structure)`
- **What:** Returns a minimal empty sheet for edge cases (empty JSON)
- **Input:** Structure string
- **Output:** The standard return shape with empty data
- **Why:** Prevents errors when opening empty JSON files

#### `castValue(displayValue, originalType)`
- **What:** Converts a spreadsheet display value BACK to its original JS type
- **Input:**
  - `displayValue` — what the user sees/typed in the cell (string or number)
  - `originalType` — the type tag from the type map
- **Output:** The properly typed value
- **Why:** This is the core of type preservation

**Conversion rules:**
```
displayValue    originalType    → output
─────────────   ────────────    ─────────
""              "string"        → ""
""              "number"        → null
""              "boolean"       → null
""              "null"          → null
""              "array"         → null
"105"           "number"        → 105
"true"          "boolean"       → true
"hello"         "string"        → "hello"
"[1,2]"         "array"         → [1, 2]
"null"          "null"          → null
"42"            "null"          → 42
```

#### `sheetDataToJson(values, typeMap, structure, headers, originalKeys)`
- **What:** Converts a 2-D value grid back into the original JSON shape
- **Input:**
  - `values` — 2-D array from FortuneSheet (row 0 = headers, row 1+ = data)
  - `typeMap` — per-cell type map from the original load
  - `structure` — `'array'` | `'object-of-objects'` | `'object'`
  - `headers` — column names
  - `originalKeys` — original top-level keys (for object-of-objects)
- **Output:** Reconstructed JSON object or array
- **Why:** This reverses `jsonToSheetData`, producing valid JSON from spreadsheet data

---

### `src/jsonValidator.js`

#### `validateJson(json, structure)`
- **What:** Checks every row against business rules
- **Input:** Reconstructed JSON + structure tag
- **Output:** Array of error objects `[{ row, key, field, message }]` (empty = all valid)
- **Why:** Catches invalid data before it's saved. See [Section 7](#7-validation-logic-detailed).

---

### `webview-src/hooks/useVscodeMessaging.js`

#### `useVscodeMessaging({ onLoadData, onValidationErrors })`
- **What:** React hook for VS Code extension ↔ webview communication
- **Input:** Object with handler functions
- **Output:** `{ postMessage }` — function to send messages to the extension
- **Why:** Encapsulates the `acquireVsCodeApi()` + event listener pattern

---

### `webview-src/components/SpreadsheetEditor.jsx`

#### `SpreadsheetEditor()` (component)
- **What:** Main React component rendering FortuneSheet
- **Key state:**
  - `sheetData` — the FortuneSheet data (set from extension messages)
  - `validationErrors` — array of current errors
  - `version` — incremented on data reload to force Workbook re-mount
- **Key refs:**
  - `skipNextChangeRef` — skips the first onChange after data load
  - `loadTimeRef` — timestamp guard (ignores onChange for 1s after load)
  - `headersRef` — column headers

#### `handleChange(data)` (callback)
- **What:** Called by FortuneSheet when any cell changes
- **Input:** Array of sheet data objects from FortuneSheet
- **Output:** Sends `{ type: 'edit', values }` message to extension
- **Why:** This is how user edits flow from the spreadsheet back to the extension

**Guard logic:**
1. Skip if `skipNextChangeRef` is true (first onChange after mount)
2. Skip if within 1 second of data load (catches extra mount-time onChange events)
3. Extract plain values from FortuneSheet's 2-D cell array
4. Send to extension

## 5. Complete Data Flow

### Loading a JSON File (Open)

```
Step 1:  User opens a .json file → VS Code reads it into a TextDocument
              │
Step 2:  VS Code calls resolveCustomTextEditor(document, webviewPanel)
              │
Step 3:  Extension creates the webview HTML → webview renders → React mounts
              │
Step 4:  Webview sends { type: 'ready' } to the extension
              │
Step 5:  Extension calls loadDocument():
              │
              ├── document.getText()  →  raw JSON string
              │
              ├── JSON.parse(text)  →  JavaScript object
              │
              ├── normalizedOriginal = JSON.stringify(json, null, 4)
              │   (saved for comparison — prevents false dirty tracking)
              │
              ├── jsonToSheetData(json)  →  { sheetData, typeMap, structure, keys, headers }
              │   │
              │   ├── Detects structure (array / object-of-objects / object)
              │   ├── Builds per-cell type map:  { "1_3": "number", "1_7": "boolean", ... }
              │   ├── Creates header row (row 0) with bold styling
              │   └── Creates data rows (row 1+) with display values
              │
              └── Sends { type: 'loadData', sheetData, headers } to webview

Step 6:  Webview receives loadData → sets sheetData state → FortuneSheet renders grid
```

### User Edits a Cell

```
Step 1:  User types in a cell and presses Enter
              │
Step 2:  FortuneSheet fires onChange(data)
              │
Step 3:  SpreadsheetEditor.handleChange():
              │
              ├── Guard check: skip if < 1 second since load
              │
              ├── Extracts plain values from FortuneSheet's 2-D cell array
              │   (ignores formatting, just gets .v or .m from each cell)
              │
              └── Sends { type: 'edit', values: [[...], [...], ...] } to extension

Step 4:  Extension receives 'edit' message:
              │
              ├── sheetDataToJson(values, typeMap, ...)
              │   │
              │   ├── Skips header row (row 0)
              │   ├── For each cell: castValue(displayValue, originalType)
              │   │   ├── "105"  + type "number"  → 105
              │   │   ├── "true" + type "boolean" → true
              │   │   ├── ""     + type "null"    → null
              │   │   └── "[1,2]" + type "array"  → [1, 2]
              │   └── Reconstructs JSON in original shape
              │
              ├── newText = JSON.stringify(json, null, 4)
              │
              ├── if (newText === normalizedOriginal) → SKIP (no real change)
              │
              ├── validateJson(json, structure)
              │   └── Returns errors array → sent to webview for banner display
              │
              ├── WorkspaceEdit.replace(document.uri, fullRange, newText)
              │   (replaces entire document with the new JSON)
              │
              ├── suppressDocChange = true  →  applyEdit  →  suppressDocChange = false
              │   (prevents the onDidChangeTextDocument listener from re-loading)
              │
              └── normalizedOriginal = newText  (update baseline for next comparison)
```

### Saving

```
Step 1:  User presses Ctrl+S
              │
Step 2:  VS Code fires onWillSaveTextDocument
              │
              └── Extension checks: any validation errors?
                  ├── YES → shows warning toast: "This file has N validation error(s)"
                  └── NO  → nothing

Step 3:  VS Code writes document.getText() to disk
              (the document already contains the latest JSON from the last edit)
```

## 6. Type Preservation System

This is the critical feature that prevents data corruption.

### The Problem

JSON has types: `100` (number), `true` (boolean), `null`, `"hello"` (string).
Spreadsheets only have cells with text. If you edit `100` to `105`, the spreadsheet sees `"105"` (a string).
If you save `"105"` back to JSON, you get `{ "price": "105" }` — **the number became a string!**

### The Solution: Type Map

When loading, we record the original type of every cell:

```javascript
typeMap = {
  "1_0":  "string",   // row 1, col 0  → _key is a string
  "1_1":  "string",   // row 1, col 1  → data_block_name is a string
  "1_4":  "null",     // row 1, col 4  → length was null
  "1_5":  "number",   // row 1, col 5  → scale was 0 (number)
  "1_7":  "boolean",  // row 1, col 7  → mandatory_in_input was false
  "1_11": "array",    // row 1, col 11 → default_value was []
  // ... one entry per cell
}
```

When saving, `castValue()` uses this map to restore types:

```
User typed "105" → typeMap says "number" → save as 105
User typed "true" → typeMap says "boolean" → save as true
User cleared a cell → typeMap says "number" → save as null
User typed "[]" → typeMap says "array" → save as []
```

### What Happens for New Rows?

If the user adds a row that wasn't in the original JSON, there's no type map entry.
The `castValue()` function defaults to `'string'` for unknown types.

## 7. Validation Logic (Detailed)

### When Does Validation Run?

Validation runs **on every cell edit** (not just on save). This gives immediate feedback.
Additionally, a VS Code warning toast appears **when saving** if there are outstanding errors.

### Schema Detection

The validator first checks if the JSON data has a `part_of_call_interface` field
in any row. If not, only the `used_in_strategy` check is applied — the detailed
checks are skipped because the JSON uses a different schema.

### The Validation Chain (matches the pseudocode exactly)

```
For each row that has "used_in_strategy":

CHECK 1:  used_in_strategy must be Y, N, true, or false
          ├── null            → ❌ ERROR
          ├── "maybe"         → ❌ ERROR
          ├── true / "y"      → proceed to CHECK 2
          └── false / "n"     → ✅ PASS (skip remaining checks)

CHECK 2:  field_name must not be empty
          (falls back to _key if field_name doesn't exist)
          ├── null or ""      → ❌ ERROR
          └── has value       → proceed to CHECK 3

CHECK 3:  data_block_category must not be empty
          (falls back to data_block_category column)
          ├── null or ""      → ❌ ERROR
          └── has value       → proceed to CHECK 4

CHECK 4:  part_of_call_interface must be Y or N
          ├── null            → ❌ ERROR
          ├── "y" / true      → proceed to CHECK 5
          └── "n" / false     → proceed to CHECK 9

  ── When part_of_call_interface = Y ──

CHECK 5:  field_type must be one of:
          boolean, string, list_string, integer, list_integer,
          decimal, list_decimal, float, list_float
          ├── null or invalid → ❌ ERROR
          └── valid           → proceed to CHECK 6

CHECK 6:  field_direction must be inout, input, or output
          (if present)
          ├── invalid         → ❌ ERROR
          └── valid or null   → proceed to CHECK 7

CHECK 7:  mandatory_presence_in_input must be Y or N
          (if present)
          ├── invalid         → ❌ ERROR
          └── valid or null   → proceed to CHECK 8

CHECK 8:  Cannot have mandatory_presence_in_input = Y
          when field_direction = output or none
          ├── violated        → ❌ ERROR
          └── ok              → ✅ PASS

  ── When part_of_call_interface = N ──

CHECK 9:  Cannot have mandatory_presence_in_input = Y
          when part_of_call_interface = N
          ├── violated        → ❌ ERROR
          └── ok              → ✅ PASS
```

### Field Name Mapping

The validator supports two naming conventions. If the "standard" field name
doesn't exist in the data, it falls back to an alternative:

| Validation Field                | Fallback Field         |
|---------------------------------|------------------------|
| `field_name`                    | `_key`                 |
| `field_data_block_category`     | `data_block_category`  |
| `field_type`                    | `datatype`             |
| `mandatory_presence_in_input`   | `mandatory_in_input`   |
| `part_of_call_interface`        | *(no fallback)*        |
| `field_direction`               | *(no fallback)*        |

## 8. Dirty Tracking & Undo/Redo

### Why Opening Must Not Dirty the File

When you open a JSON file, FortuneSheet fires an `onChange` event during its internal
initialization. Without protection, this would:
1. Trigger the edit handler
2. Reconstruct the JSON with `JSON.stringify(json, null, 4)` (different indentation from original)
3. Apply a WorkspaceEdit (different text = "modification")
4. Mark the document as dirty
5. Prompt the user to save when they close

### Three-Layer Protection

1. **Webview guard:** `skipNextChangeRef` skips the first `onChange` after data load
2. **Timestamp guard:** Ignores all `onChange` within 1 second of data load
3. **Extension comparison:** `normalizedOriginal` comparison — if the reconstructed JSON
   is identical to the parsed-and-re-stringified original, the edit is silently dropped

### How Undo/Redo Works

```
User makes edit → WorkspaceEdit applied → VS Code records undo entry
User presses Ctrl+Z → VS Code reverts the document
→ onDidChangeTextDocument fires (suppressDocChange is false)
→ loadDocument() re-reads document, sends new data to webview
→ Webview re-renders FortuneSheet with version key increment
→ FortuneSheet fires onChange on mount → skipped by guards
```

## 9. Save Lifecycle

```
1. User presses Ctrl+S (or File → Save)
2. VS Code fires onWillSaveTextDocument
3. Extension checks currentValidationErrors:
   - If errors exist → shows warning toast
   - If no errors → nothing
4. VS Code writes document.getText() to disk
5. The document is now clean (not dirty)
```

**Important:** The file is always saved. Validation errors produce a WARNING,
not a block. This is intentional — the user is informed but not prevented from saving.

## 10. Toolbar Customization Guide

### Current Toolbar Configuration

In `SpreadsheetEditor.jsx`, the `TOOLBAR_ITEMS` array controls which buttons appear:

When you want to integrate this feature into a larger existing extension,

### Files to Copy

```
MUST COPY:
├── src/jsonEditorProvider.js      ← core editor provider
├── src/jsonConverter.js           ← data conversion + type map
├── src/jsonValidator.js           ← validation rules
├── webview-src/                   ← entire folder (React webview source)
│   ├── main.jsx

│   └── utils/themeHelper.js
├── dist/webview/bundle.js         ← pre-built webview bundle

DO NOT COPY:
├── package.json                   ← merge dependencies into target's package.json
├── src/extension.js               ← merge registration into target's activate()
├── node_modules/                  ← reinstall in target project
```

### Execution Flow (Step by Step)

```

 1. VS Code opens .json file
    │
 2. activate() → JsonEditorProvider.register()
    │
 3. resolveCustomTextEditor(document, webviewPanel)
    │
    ├── Creates webview HTML (bundle.js loads React + FortuneSheet)
    │
    ├── Webview sends: { type: 'ready' }
    │
    ├── Extension calls loadDocument():
    │   ├── Parses JSON
    │   ├── Builds typeMap + sheetData
    │   ├── Saves normalizedOriginal
    │   └── Sends { type: 'loadData', sheetData, headers }
    │
    ├── FortuneSheet renders the grid
    │   └── onChange fires (mount) → SKIPPED by guards
    │
 4. User edits cell "true" → "false"
    │
    ├── FortuneSheet onChange fires
    │   ├── Guard: skip? NO (past 1s window)
    │   ├── Extracts values from 2-D array
    │   └── Sends { type: 'edit', values }
    │
    ├── Extension receives 'edit':
    │   ├── sheetDataToJson(values, typeMap, ...)
    │   │   └── castValue("false", "boolean") → false
    │   ├── JSON.stringify(json, null, 4)
    │   ├── Compare with normalizedOriginal → DIFFERENT
    │   ├── validateJson(json, structure) → [] (no errors)
    │   ├── WorkspaceEdit.replace(document, newText)
    │   └── Document is now DIRTY (modified dot appears on tab)
    │
 5. User presses Ctrl+S
    │
    ├── onWillSaveTextDocument fires
    │   └── No validation errors → no warning
    │
    └── VS Code writes document to disk → tab becomes clean
```

## Quick Reference Card

| Action                      | What Happens                                                  |
|-----------------------------|---------------------------------------------------------------|
| Open .json                  | Parsed → spreadsheet grid, NO document modification           |
| Edit cell                   | FortuneSheet → values → extension → type-cast → update doc    |
| Save (Ctrl+S)               | Warning if validation errors, then writes to disk             |
| Undo (Ctrl+Z)               | VS Code reverts document → webview re-renders with old data   |
| Close without saving        | VS Code asks only if document was actually modified            |
| Empty cell + number type    | Saved as `null`                                               |
| Empty cell + string type    | Saved as `""`                                                 |
| Invalid JSON file           | Shows "Invalid JSON" error in webview                         |
